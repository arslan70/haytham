Write the full markdown content for a single authentication story. Auth stories cover auth provider setup, login/register flows, middleware, route protection, session management, and password reset.

## STORY CONTEXT

You will receive:
1. The story skeleton (id, title, layer, implements, depends_on, summary)
2. The shared project context (system goal, MVP scope, capability model, architecture decisions, build/buy analysis)
3. A compact list of ALL story skeletons for cross-referencing dependencies

## CONTENT FORMAT

Write the `content` field as markdown. Include these sections as appropriate:

```
## Description
Brief description of what this story accomplishes and why.

## Files to Create
- `/path/to/file.ext` - Purpose of this file

## Acceptance Criteria

```gherkin
Scenario: Successful registration
  Given I am on the registration page
  When I submit valid email and password
  Then I should be redirected to dashboard
  And my account should be created in the database

Scenario: Failed login
  Given I am on the login page
  When I submit incorrect credentials
  Then I should see an error message
  And I should remain on the login page
```

## Required Permissions
- CREATE on [entity] (condition)
- READ on [entity] (condition)
```

## RULES

1. **NO IMPLEMENTATION CODE** — Stories are specifications, not implementations
2. **GHERKIN FORMAT** — Use Gherkin scenarios for acceptance criteria
3. **COVER ERROR CASES** — Include scenarios for invalid input, unauthorized access, expired sessions
4. **MIDDLEWARE SPEC** — If this story covers route protection, specify which routes are protected and how
5. **ADAPT TO STACK** — Use the auth provider from architecture decisions. Do NOT assume any specific provider
6. **PERMISSION TRACEABILITY** — For every write operation in acceptance criteria, list the required permission
